## P0 — Launch-critical path (do FIRST) 🔴

> The pivot removed **none** of these. The app is Google-OAuth-only and the headline feature is Gmail import, so the one question that gates Sunday is:
> **"Can a brand-new invited person open the Vercel URL, sign in with Google, and complete a Gmail import end-to-end?"**
> Everything below is on the path to that "yes."

- [ ] **Privacy policy live at a public URL** — data collected, retention, deletion, Google Limited Use disclosure. Blocks the OAuth consent screen. *(verify a live route exists + is reachable unauthenticated)*
- [ ] **VERIFY — 30 waitlisters added as Google OAuth test users** in Google Cloud Console (test-mode cap = 100). *Cannot be checked from the repo — confirm in Console.*
- [ ] **VERIFY — Gmail OAuth verification status** — in test mode, refresh tokens expire ~7 days; set onboarding expectations if still unverified. *Confirm current Google policy + consent-screen state in Console.*
- [ ] **Onboarding copy for the "Google hasn't verified this app" warning** — testers will hit it; tell them it's expected and how to click through ("Advanced → Go to Nothing To Wear").
- [ ] **CSP/HSTS verified on the deployed Vercel URL** — confirm Google OAuth still completes on the real deployment, not just localhost.
- [ ] **Smoke test the full path on the deployed URL** from a fresh Google account: sign in → land in app → run a Gmail import → item appears in closet.
- [ ] **Supabase backups / PITR enabled** + one restore test (30 real users = real data).

## Feature 4 · Email import (Gmail) — **headline, needs the most UX work** 🔴

> "Search is basically done, Care I'm happy with — but the UX of email ingestion needs improvement." This is where testers form their first impression of the magic.

- [ ] **Loading / progress states** — searching Gmail, parsing, and importing each need clear feedback (no dead-air freezes). A frozen-looking import is the #1 thing that will read as "broken."
- [ ] **Empty / no-results state** — friendly message + what to try (adjust date range, keywords) instead of a blank screen.
- [ ] **Import review clarity** — make the "N items skipped — not clothing" and edit-before-save flow obvious and forgiving.
- [ ] **Error states** — expired token, no Gmail permission, network fail → plain-language recovery, not a stack trace.
- [ ] Instrument the funnel (see Observability): `import_started` → `import_results_shown` → `import_finished` with counts, so you can see exactly where testers drop.

**Testing criteria**
- [ ] Import from a real Gmail account end-to-end; every wait shows progress
- [ ] Zero-results and error paths show friendly, actionable copy
- [ ] Funnel events fire and are visible in PostHog

## Observability & feedback (PostHog + Sentry) — wire BEFORE inviting anyone 🔴

> `src/lib/monitoring.ts` already scaffolds both, consent-gated. What's missing is an **event taxonomy, user identify, feature flags, and a feedback channel.** This is how you answer "who is using what" — testers *say* they love Search; the data tells you if 4% actually used it.

- [ ] **`identify()` the signed-in user** in PostHog (uses `person_profiles: "identified_only"` already set) so events tie to a person.
- [ ] **Event taxonomy** — capture: `account_created`, `import_started`, `import_results_shown`, `import_finished` (with item count), `item_added` (manual vs import), `item_edited`, `item_deleted`, `search_used`, `filter_used`, `care_guide_opened`, plus session length.
- [ ] **Session replay** (PostHog) on — watch the "import froze" reports instead of guessing.
- [ ] **Sentry** confirmed capturing JS errors on the deployed build (release/version tagged — see Versioning).
- [ ] **In-app feedback button** → writes to a Supabase `feedback` table (message + auto-attached: user id, app version, browser, screen size, current view). Optional screenshot to Supabase Storage.
- [ ] **Feature flags** — two backends by purpose:
  - **Supabase profile column** for **core-UI gates** like status/location (works even if a user *declines* analytics consent — fails closed / stays dark).
  - **PostHog flags** for genuinely-optional experiments only.

**Testing criteria**
- [ ] Sign in → PostHog shows an identified person with events
- [ ] Trigger an error → it appears in Sentry with the app version
- [ ] Submit feedback → row lands in Supabase with metadata
