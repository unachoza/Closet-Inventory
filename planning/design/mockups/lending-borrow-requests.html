E4 Part Deux · rev 2 · Design Spec · 2026-07-10 · Post-MVP

Lending via borrow requests

Nothing To Wear · request → approve → return → condition
Replaces owner-recorded simple lending (rev 1)
Notifications: activity-notifications.html

Every borrow is a request. The borrower asks in one tap; the owner approves with care instructions prefilled from the item's care data and a due date; the borrower marks the return (clean, or needs laundering); the owner confirms the condition it actually came back in — and that condition updates the item's status automatically. All events land in the calm Activity tab of the profile — no bells, no push, no noise.

i.Every borrow is a request

Rev 2 replaces owner-recorded lending: the borrower must be a signed-in viewer (Part Une makes that free). One state machine — pending → approved/denied → returned_pending → closed — with both parties' actions logged.

ii.Care terms are the contract

Prefilled from the item's care data at approval, editable and augmentable, visible to both for the loan's duration, acknowledged in one tap. US-4.5 delivered inside the approve flow — no separate ceremony.

iii.Two honest checkpoints beat one

The borrower declares (clean / needs laundering), the owner confirms (six condition chips + note). Disagreement is recorded, never litigated — this is a trusted circle, and the history is the memory.

iv.Condition drives truth

The owner's condition log auto-maps to status — clean, dirty, or in_repair — so confirming the return and updating the closet are one gesture. Supersedes the old return_loan → clean transition.
