## Steps to unblock (you must do these)

### 1. Add `VITE_SUPABASE_ANON_KEY` to `.env`

In the [Supabase dashboard](https://supabase.com/dashboard/project/rawuntspvetfdtrqggen/settings/api):  
`Project Settings → API → Project API keys → anon / public`

Copy that value and add to `.env`:
```
VITE_SUPABASE_ANON_KEY=<paste anon key here>
```

### 2. Enable Google provider in Supabase

In the dashboard:  
`Authentication → Providers → Google`

- Enable it
- Paste your **Google OAuth Client ID** (already in `.env` as `VITE_GOOGLE_CLIENT_ID`):  
  `18124693407-qnkbe2o17q6i4iastetgj5o6fjlfv0ho.apps.googleusercontent.com`
- Paste your **Google OAuth Client Secret** (from Google Cloud Console)

### 3. Add Supabase callback URI to Google Cloud Console

In [Google Cloud Console](https://console.cloud.google.com/) → Credentials → your OAuth 2.0 Client:

Under **Authorized redirect URIs**, add:
```
https://rawuntspvetfdtrqggen.supabase.co/auth/v1/callback
```

This is the URI that was causing `redirect_uri_mismatch`. Supabase handles the callback and redirects back to `window.location.origin` (your app).

> ⚠️ **`gmail.readonly` scope note:** Google classifies this as a **sensitive scope**. For development, your app can stay in "Testing" mode in the OAuth consent screen. For production, you need Google's [CASA Tier 2 verification](https://support.google.com/cloud/answer/9110914). The spike can be proved in Testing mode with your own Google account.

### 4. Verify the spike works

After the above is done:

1. Temporarily wire `SupabaseAuthProvider` into `App.tsx` (see below)
2. Add a test button that calls `signIn()` from `useSupabaseAuthContext`
3. Click it — you should be redirected to Google consent and back
4. After redirect, `session.provider_token` should be a Google access token
5. Make a test call to `https://gmail.googleapis.com/gmail/v1/users/me/messages` with that token — if it returns 200, the spike is proved

Minimal wiring for proof (add temporarily to `App.tsx`):
```tsx
import { SupabaseAuthProvider } from "./context/SupabaseAuthContext";
// wrap existing providers:
<SupabaseAuthProvider>
  {/* existing providers */}
</SupabaseAuthProvider>
```

### 5. Key finding to capture

After the spike succeeds, document in `planning/epics/E1-cloud-backend.md`:
- Whether `provider_token` survives a hard page reload (expected: NO — Supabase doesn't persist the Google token across reloads, only the Supabase JWT)
- If it doesn't survive: E1-1.3 needs a Supabase Edge Function to refresh it from `provider_refresh_token`
- The integration path: one Google sign-in → both app auth (Supabase session) + Gmail access (provider_token)

## What this replaces (eventually)

The existing `@react-oauth/google` popup (`useGmailAuth` + `GmailAuthContext`) stays in place until the spike is proved. Once proved, the migration is:

1. `GmailImport` calls `useSupabaseAuthContext().gmailAccessToken` instead of `useGmailAuthContext().accessToken`
2. The `@react-oauth/google` dependency and its `GoogleOAuthProvider` wrapper can be removed
3. One sign-in covers both app auth and Gmail access — no second popup

This migration is E1-2.x, not part of this spike.
