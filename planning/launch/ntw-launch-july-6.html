ClosetRepository interface with three implementations (local, Supabase, synced) is textbook. The SyncedClosetRepository correctly implements offline-first writes (local first, then async to Supabase) with last-write-wins reconciliation via updatedAt. This is a pattern many production apps get wrong.

Concerns

Critical Fire-and-Forget Async Writes Silently Lose Data
SyncedClosetRepository uses void this.remote.add(stamped).catch(...) for remote writes. If a remote write fails, the failure is recorded in syncFailureTracker but never retried. There's no retry queue, no exponential backoff, and no mechanism to resurface failed writes on the next getAll(). A user who adds items while briefly offline will see them locally but they'll silently never reach Supabase. The reconcile in getAll() only runs on sign-in / userId change — not periodically.
src/services/syncedClosetRepository.ts:94-99
Critical No Router — View State is Fragile
Navigation is managed by a view string in React Context with no URL integration. Browser back button does nothing. Deep-linking is impossible. Refreshing the page always resets to the carousel. For a production app targeting 30 real users, this will create constant friction. At minimum, consider syncing view to the URL hash before launch.
src/context/ViewContext.tsx, src/App.tsx
Concern Unsafe Type Casts in DB Layer
Several fields use as unknown as Json or as ClothingItem["field"] casts in rowToItem and the insert/update mappers. These bypass type checking at the one boundary where runtime data enters the app. If the DB schema drifts (e.g., a status value is added to Postgres but not to ItemStatus), these casts will silently produce invalid objects. Consider runtime validation (Zod or a simple asserter) for the rowToItem boundary.
src/services/supabaseClosetRepository.ts:26-51
Concern upsertMaterials Deletes Then Inserts Without a Transaction
upsertMaterials does a DELETE followed by INSERT for item materials. If the insert fails (network drop, constraint violation), the materials are gone with no rollback. Should use Supabase's RPC or a database function to wrap this in a transaction. Low probability but high impact on data integrity.
src/services/supabaseClosetRepository.ts:129-141
Concern Three Icon Libraries
The project ships Heroicons, Phosphor Icons, and Lucide React. Each adds ~20–50KB to the bundle (tree-shaking helps, but three separate SVG systems means three render paths and inconsistent visual weight). Standardize on one before launch to reduce bundle size and visual inconsistency.
package.json
Concern Hardcoded UI Lists Create Silent Bugs
Your own planning docs flag this correctly: category lists, filter dimensions, status options, and location registries are hardcoded arrays that tsc won't catch when you add a new value. This is a recurring maintenance risk. Consider deriving the filter options from the type union at build time, or at minimum adding an exhaustive-check utility test that fails when the type and array drift.
src/utils/constants.ts, src/hooks/useClosetFilters.ts
Concern GmailSpike.tsx Still in Source
The Gmail OAuth spike file is still present and importable. It served its purpose (verified the flow works); leaving it in the tree risks someone importing it instead of the production path, and it may contain exploratory patterns that bypass security measures. Remove or move to a /spikes archive before launch.

Strategic Recommendations

1. Start Block A now, not after Block B is "done." Security is the hard launch gate. Every week spent deepening the inventory spine before dev/prod split and account deletion is a week the 30-person waitlist doesn't have access. The inventory work you've done is solid — ship it with filters + legend + quick-actions, but don't chase P1-6/7/9/11 before security.

2. Strongly consider Option B (drop full PWA for now). Your waitlisters can use the responsive web app today. The PWA manifest + service worker can ship 1–2 weeks after launch as a fast-follow. This saves ~10 dev-days and pulls the timeline to ~3–4 weeks. The "fix mobile rendering" half of Block C (bottom nav, tap targets, FAB) still ships.

3. Submit OAuth verification Gate 1 immediately. The review queue is 1–3 months. Test mode unblocks the beta, but you want the verification pipeline moving in parallel. The only dependency is the privacy policy URL, so write the policy first.

4. Address the sync retry gap before launch. Fire-and-forget remote writes mean offline edits silently disappear. This is a data integrity issue your early users will hit and lose trust over. A simple IndexedDB-backed retry queue is ~1.5 days and pays for itself in user trust.

5. Add URL routing before users touch it. No back button, no deep links, no refresh persistence — this will generate more support friction than any missing feature. Hash-based routing is a 1.5-day investment that prevents a class of "the app is broken" reports.
