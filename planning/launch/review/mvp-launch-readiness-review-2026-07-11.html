Newly Surfaced — hidden bugs that must be fixed before the 30-user beta

Critical · S effort

Fuse.js index rebuilt twice per interaction

src/hooks/useFuzzySearch.ts:47-73 closes over debouncedQuery only, so new Fuse(...) runs on every items mutation, and EntireClosetView invokes both searchResults and getMatchKeys per render — two full re-indexes per filter/search change. At 300+ items iPhone SE will visibly stall.

Critical · S effort

Silent QuotaExceededError on signed-out writes

src/utils/safeStorage.ts:11-19 catches quota errors and only logs a warning. Signed-out users see a green success toast while their new item was not persisted. Safari's ~5 MB localStorage ceiling hits at ~20 base64 photos. Data-loss UX bug.

High · S effort

CSP is skeletal, not defense-in-depth

vercel.json sets only script-src, object-src, base-uri. No default-src, connect-src, img-src, style-src, frame-ancestors. Ship it right before waitlisters land — test in report-only mode first so Supabase + Google OAuth still resolve.

Med · S effort

Docs drift: README + ARCHITECTURE describe deleted hooks

useLocalCloset, useLocalStorageCloset, and a duplicate inferProductAttributes.ts are all referenced in docs but no longer exist in src/. ClothingItem's "index signature" warning is stale. Onboarding-hazard for the future dev partner.

Med · S effort

Two write paths to the same localStorage key

useCloudCloset.ts:21-30's seed() writes my_closet_key directly, and localClosetRepository.ts:28-30 writes the same key. Race on first-mount reconcile. Route seed through repo.

Med · M effort

Clock-skew data loss in sync

Last-write-wins uses client-set updatedAt. A device with a fast clock will overwrite a device that saved later. Server-side updated_at trigger exists; make it authoritative.

§ iv

Engineering Review

performance · scalability · maintenance

Strengths

  • Clean repository seam — services/closetRepository with Local and Synced implementations. Hooks never touch localStorage or the DB client directly.
  • Offline-first sync design — last-write-wins on updatedAt, background push, syncFailureTracker surfaces in NavBar, verified in an E2E PWA suite (offline-closet.pwa.spec.ts).
  • Type safety end-to-end — @ntw/types workspace package; TS strict mode with noUnusedLocals, noUnusedParameters, noFallthroughCasesInSwitch.
  • Storage pipeline is right — client-side compression (≤1200px, JPEG q0.8), Storage upload, path-only in row, signed URL at render with auto-refresh. Server-side bucket MIME/size enforcement live.
  • Testing culture — 1,300+ Vitest specs + 14 Playwright suites; regression tests were added for every bug-review finding. This is not typical of solo builds.

Weaknesses / Bottlenecks

  • Lighthouse Perf 55 — single 291 KiB JS chunk on first paint, ~200 KiB unused. LCP 7.1s on throttled mobile. Route-level dynamic import() is the #1 fix.
  • Three icon libraries — @heroicons/react, @phosphor-icons/react, lucide-react. Pick one; drop the others (~40-60 KB tree-shakeable).
  • Fuse.js index rebuild on every closet mutation — for 500+ item closets this becomes noticeable. Memoize the index by items.length + updatedAt max.
  • base64 in localStorage for signed-out photo storage — Safari 5 MB quota fails silently. Bounded now by the guard, but no per-item size ceiling in the client.
  • Sync retry queue deferred — offline write that never reconciles (tab closes before reload) stays stale in cloud. Accepted risk in E1-1.6.
  • No indexed access patterns in RLS SELECTs for future status/location_id filters. Add btree on (user_id, status) before item counts get real.

Scalability outlook

The offline-first architecture is the right long-term choice — writes are instant, network failures degrade gracefully, and RLS makes multi-tenant safety straightforward. What breaks at scale isn't the design, it's the client-side everything: Fuse index rebuilds, full-closet re-sorts, and localStorage-as-cache for 5,000-item power users. Instrument first — real p95 numbers from the 30-user beta will separate the theoretical from the actual bottleneck. The Supabase Postgres side is not the risk; the browser is.

Moat · Uniqueness · Competitive Read

what to defend · what to cut
The closet that tells you the truth about itself. — proposed one-liner, from PRODUCT_VISION_2026-06-20.md

What is the moat?

  • Status + Location as core primitives — no competitor treats "where is my sweater right now" as first-class product data. It's the uncompeted core.
  • Frictionless fill from Gmail back-catalog — competitors ask you to photograph 300 items. NTW parses years of receipts. This is the acquisition wedge.
  • FashionParser inference engine — 17 attribute maps, 6 normalizers, 6 inference fns. Domain-specific knowledge that grows with each retailer added.
  • Borrow / share loop (post-MVP) — the only feature with a native viral mechanic. Every "can I see your closet?" is a warm referral.

What is table stakes?

  • Grid view, cards, photos — everyone has this.
  • Fuzzy search, filters, sort — table stakes but well-executed.
  • Multi-material composition bar — pretty, not defensible.
  • Fabric care content — nice differentiator, not defensible alone.
  • Outfit builder / weather / packing — competitors compete here; NTW deliberately doesn't.

vs. Whering / Stylebook / Acloset / OpenWardrobe

  • Whering · outfit + style AI. Better: polished mobile app, AI outfit suggestions. NTW better: inventory-truth layer, Gmail auto-fill, no styling AI required.
  • Stylebook · manual catalog + calendar. Better: 12+ years of polish, packing lists, calendar. NTW better: no manual photography chore, status/location tracking, cloud sync.
  • Acloset · AI stylist + community. Better: AI generation, look feeds. NTW better: privacy-first, receipts-import, actual logistics data.
  • OpenWardrobe · open-source purist. Better: transparency, self-host. NTW better: hosted UX, retailer parsing, launch-ready polish.

If you cut 50% of the roadmap

Keep: Inventory (E3 fill), Location (E2 spine), Status (E2 spine), Care (E8 content), the friend-view share link (E4 Part Une — the growth loop).

Cut: outfit builder (E6), insights/viz (E7), travel packing (E9), monetization scaffolding (E10), laundry forecast (E11), i18n (E13), full profile hub (E12 — keep only locations).

The docs already point this direction. The MVP shipping today is the 50% cut. That's the discipline win.
